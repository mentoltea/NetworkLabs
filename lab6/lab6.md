## 2. Теоретические сведения

### 2.1 Надёжность передачи данных

Несмотря на использование TCP, который гарантирует доставку байтов, прикладной уровень может столкнуться с:

- потерей логических сообщений;
- дублированием сообщений;
- нарушением целостности данных;
- задержками и нестабильным временем отклика.

Для решения этих проблем используются механизмы:

- подтверждение доставки (ACK);
- повторная отправка (retransmission);
- контроль целостности;
- дедупликация сообщений.
---
### 2.2 Время отклика (RTT)

RTT (Round Trip Time) — время от отправки сообщения до получения ответа.

```cpp
RTT = t_receive - t_send
```
---
### 2.3 Джиттер (Jitter)

Джиттер — изменение задержки между последовательными пакетами:

```cpp
Jitter = |RTT_n - RTT_{n-1}|
```
---
### 2.4 Потери пакетов (Loss)

Процент потерянных сообщений:

```cpp
Loss = (sent - received) / sent * 100%
```
---
### 2.5 Подтверждение доставки (ACK)

Каждое сообщение должно подтверждаться:

```cpp
Sender → Message(id)  
Receiver → ACK(id)
```

Если подтверждение не получено — выполняется повторная отправка.

---
### 2.6 Дедупликация

При повторной отправке возможно получение дубликатов.

Решение:

- хранить обработанные `msg_id`;
- игнорировать повторные сообщения.
---

В рамках работы требуется реализовать:

2. **Эмуляцию сетевых помех**

3. **Надёжную доставку сообщений**

В результате выполнения работы система должна корректно функционировать в условиях нестабильной сети и обеспечивать доставку сообщений с учётом реализованных механизмов.

## 4. Расширение протокола

Добавить новый тип сообщения:

```cpp
enum {
    MSG_HELLO        = 1,
    MSG_WELCOME      = 2,
    MSG_TEXT         = 3,
    MSG_PING         = 4,
    MSG_PONG         = 5,
    MSG_BYE          = 6,

    MSG_AUTH         = 7,
    MSG_PRIVATE      = 8,
    MSG_ERROR        = 9,
    MSG_SERVER_INFO  = 10,

    MSG_LIST         = 11,
    MSG_HISTORY      = 12,
    MSG_HISTORY_DATA = 13,
    MSG_HELP         = 14,

    MSG_ACK          = 15
};
```

### Назначение MSG_ACK

Сообщение типа `MSG_ACK` используется для подтверждения успешного получения и обработки сообщения на стороне получателя.

Особенности:

- содержит `msg_id` подтверждаемого сообщения;
- отправляется сервером после обработки сообщения клиента;
- используется клиентом для определения факта доставки;
- при отсутствии `MSG_ACK` клиент инициирует повторную отправку.

## 5. Монитор задержек, джиттера и потерь
В данном задании необходимо реализовать механизм диагностики сети между клиентом и сервером.

---
### 5.1 Требования к клиенту

Добавить команду:

```cpp
/ping  
/ping N
```

- `/ping` — отправляет 10 запросов (по умолчанию);
- `/ping N` — отправляет N запросов.

Каждый запрос должен иметь уникальный `msg_id`.

---
### 5.2 Отправка запросов (клиент)

Клиент должен:

1. Сгенерировать `msg_id`;
2. Зафиксировать время отправки (`timestamp`);
3. Отправить сообщение `MSG_PING` на сервер;
4. Сохранить время отправки для последующего расчёта RTT.

Важно:

- все отправленные `msg_id` должны храниться;
- необходимо уметь определить, на какой запрос пришёл ответ.

---

### 5.3 Обработка на сервере

Сервер при получении `MSG_PING` должен:

1. Не выполнять дополнительной логики;
2. Немедленно отправить `MSG_PONG`;
3. Передать тот же `msg_id` обратно клиенту.

---

### 5.4 Обработка ответов (клиент)

При получении `MSG_PONG` клиент должен:

1. Найти соответствующий `msg_id`;
2. Вычислить RTT:
    - текущее время − время отправки;
3. Вычислить jitter:
    - разницу между текущим RTT и предыдущим;
4. Зафиксировать успешный ответ.

Если ответ не пришёл за заданное время (например, 1–2 секунды):

- считать запрос потерянным.

---

### 5.5 Команда `/netdiag`

После завершения всех запросов клиент должен вычислить:

- **RTT avg** — среднее значение RTT;
- **Jitter avg** — среднее значение джиттера;
- **Loss** — процент потерянных сообщений.

Клиент выводит:

```cpp
RTT avg : X ms  
Jitter  : Y ms  
Loss    : Z %
```

Дополнительно сохраняет данные в файл:

```cpp
net_diag_<nickname>.json
```
Формат может быть произвольным, но должен содержать рассчитанные метрики.

---

### 5.6 Логи сервера

```cpp
[Transport][PING] recv MSG_PING (id=3)  
[Transport][PING] send MSG_PONG (id=3)  
[Application] client=alice, rtt=12.8ms, loss=0%
```

---

### 5.8 Ожидаемое поведение

Пример работы клиента:

```cpp
> /ping 5  
  
PING 1 → RTT=4.2ms    
PING 2 → RTT=5.1ms | Jitter=0.9ms    
PING 3 → RTT=12.8ms | Jitter=7.7ms    
PING 4 → timeout    
PING 5 → RTT=6.3ms | Jitter=0.0ms    
```
  
```cpp
> /netdiag  
  
RTT avg : 7.1 ms    
Jitter  : 2.8 ms    
Loss    : 20.0%
```

## 6. Эмуляция сетевых помех

В данной части работы необходимо добавить на сервер учебную эмуляцию нестабильной сети. Это позволит проверить, как клиент и сервер работают при задержках, потерях и повреждении данных.

### 6.1 Параметры запуска сервера

Сервер должен поддерживать дополнительные параметры командной строки:
```cpp
--delay=100  
--drop=0.2  
--corrupt=0.1
```

где:

- `--delay=100` — искусственная задержка обработки сообщения в миллисекундах;
- `--drop=0.2` — вероятность потери сообщения;
- `--corrupt=0.1` — вероятность повреждения данных сообщения.

Если параметры не указаны, сервер должен работать в обычном режиме без помех.

---

### 6.2 Общая логика работы

После получения данных от клиента сервер должен выполнить проверку на наличие сетевых помех **до передачи сообщения в прикладной уровень**.

То есть между `recv()` и обработкой сообщения должна находиться специальная функция эмуляции, которая может:

- задержать обработку;
- отбросить сообщение;
- изменить часть данных.

---

### 6.3 Искусственная задержка

Если задан параметр `--delay`, сервер должен перед обработкой сообщения сделать паузу:

```cpp
usleep(delay * 1000);
```

Это имитирует медленную сеть или задержку на промежуточном узле.

Например, при `--delay=100` каждое сообщение будет обрабатываться с задержкой 100 мс.

---

### 6.4 Потеря сообщения

Если задан параметр `--drop`, сервер должен с заданной вероятностью полностью игнорировать входящее сообщение.

В этом случае:

- сообщение не передаётся в прикладной уровень;
- сервер не отвечает на него;
- клиент считает, что сообщение потеряно.

Например, при `--drop=0.2` примерно 20% сообщений не должны обрабатываться.

---

### 6.5 Повреждение данных

Если задан параметр `--corrupt`, сервер должен с заданной вероятностью изменить один случайный байт в `payload`.

Это позволяет имитировать повреждение данных при передаче.

Требования:

- изменяется только один байт;
- изменение выполняется случайным образом;
- после изменения сообщение продолжает обрабатываться.

---

### 6.6 Порядок применения помех

Если включено несколько механизмов одновременно, их рекомендуется применять в следующем порядке:

1. задержка;
2. потеря;
3. повреждение.

То есть сервер:

1. получает данные;
2. при необходимости делает задержку;
3. проверяет, нужно ли отбросить сообщение;
4. если сообщение не отброшено — может повредить данные;
5. передаёт результат в прикладной уровень.

---

### 6.7 Логирование

Каждый случай применения помехи должен быть явно записан в лог сервера.

Примеры:

```cpp
[Transport][SIM] DROP (id=41, rate=0.2)  
[Transport][SIM] DELAY applied: 150 ms  
[Transport][SIM] CORRUPT payload (id=42)
```

Если сообщение было потеряно, дальнейшие логи прикладного уровня для него появляться не должны.

---
### 6.8 Требования к реализации

Необходимо соблюдать следующие условия:

- эмуляция выполняется на сервере;
- эмуляция выполняется до прикладной обработки;
- стандартный TCP стек операционной системы не изменяется;
- прикладной уровень не должен знать, почему сообщение не дошло;
- механизм должен использоваться для проверки работы повторной отправки и ACK.

---

### 6.9 Ожидаемый результат

При запуске сервера с параметрами помех студент должен увидеть, что:

- часть сообщений обрабатывается с задержкой;
- часть сообщений теряется;
- часть сообщений приходит с изменёнными данными;
- клиент в ответ на потерю запускает повторную отправку.

## 7. Повторная отправка и защита от дубликатов

### 7.1 Требования к клиенту

При отправке сообщений:

- `MSG_TEXT`
- `MSG_PRIVATE`
- `MSG_PING`

клиент должен:

1. Сохранить сообщение в очередь:

```cpp
typedef struct {  
    MessageEx msg;  
    time_t send_time;  
    int retries;  
} PendingMsg;
```

2. Ожидать ACK:

- таймаут: 2 секунды;
- максимум 3 попытки.

---

### 7.2 Алгоритм повторной отправки

если ACK не получен:  
    повторить отправку  
    увеличить retries  
если retries > 3:  
    считать сообщение недоставленным

---

### 7.3 Логи клиента

```cpp
[Transport][RETRY] send MSG_TEXT (id=41)  
[Transport][RETRY] wait ACK timeout  
[Transport][RETRY] resend 1/3 (id=41)  
[Transport][RETRY] ACK received (id=41)
```

---

### 7.4 Требования к серверу

Сервер должен хранить для каждого клиента:

```cpp
uint32_t last_ids[32];
```

---

### 7.5 Алгоритм обработки

При получении сообщения:

1. Проверить `msg_id`:
    - если уже был → игнорировать;
2. Если новый:
    - обработать сообщение;
    - сохранить `msg_id`;
    - отправить `MSG_ACK`.

---

### 7.6 Логи сервера

```cpp
[Application][ACK] process MSG_TEXT (id=41)  
[Transport][ACK] send MSG_ACK (id=41)  
[Application][DEDUP] duplicate ignored (id=41)
```

---

### 7.7 История сообщений

В JSON-файл истории записываются:

- только уникальные сообщения;
- только успешно обработанные (`delivered=true`).

Попытки повторной отправки не сохраняются.

## 8. Общие требования

### 8.1 Логирование

Все реализованные механизмы должны сопровождаться логированием по модели TCP/IP. Это необходимо для того, чтобы было видно, на каком уровне происходит обработка сообщения и какие дополнительные механизмы были задействованы.

В логах должны использоваться следующие уровни:

```cpp
[Application]  
[Transport]  
[Internet]  
[Network Access]
```

Требования:

- логирование обязательно для входящих сообщений;
- логирование обязательно для исходящих сообщений;
- при использовании новых механизмов в логе должна быть дополнительная пометка;
- записи должны быть понятными и позволять проследить полный путь обработки сообщения.

Для новых механизмов необходимо использовать отдельные префиксы:

```cpp
[Transport][PING]  
[Transport][SIM]  
[Transport][RETRY]  
[Transport][ACK]
```
где:

- `[Transport][PING]` — обработка диагностических сообщений `MSG_PING` и `MSG_PONG`;
- `[Transport][SIM]` — эмуляция сетевых помех;
- `[Transport][RETRY]` — повторная отправка сообщения;
- `[Transport][ACK]` — подтверждение доставки сообщения.

Пример логов:

```cpp
[Transport][PING] recv MSG_PING (id=5)  
[Transport][SIM] DROP (id=5, rate=0.2)  
[Transport][RETRY] resend 1/3 (id=5)  
[Transport][ACK] send MSG_ACK (id=5)  
[Application] process MSG_TEXT (id=41)
```

---

### 8.2 Потокобезопасность

Так как сервер и клиент работают с несколькими потоками, общие структуры данных должны быть защищены от одновременного доступа.

Необходимо обеспечить потокобезопасную работу со следующими данными:

- очередью ожидающих подтверждения сообщений `PendingMsg`;
- буфером последних обработанных `msg_id`;
- файлом истории сообщений;
- другими общими структурами, если они используются одновременно из нескольких потоков.

Для синхронизации необходимо использовать:

pthread_mutex

Требования:

- перед изменением общей структуры должен выполняться `pthread_mutex_lock()`;
- после завершения работы с общей структурой должен выполняться `pthread_mutex_unlock()`;
- не допускаются состояния гонки при повторной отправке, обработке ACK и записи истории.

В результате выполнения лабораторной работы студент должен предоставить:

- исходный код клиента и сервера;

    - потеря сообщения;
    - повторная отправка;
    - получение ACK;
    - успешная доставка;
- краткий отчёт по выполненной работе.

Отчёт может быть небольшим, но должен содержать:

- цель работы;
- описание реализованных механизмов;
- пример тестирования при включённых помехах;
- вывод о работе механизма повторной отправки и подтверждения доставки.

## 9. Ожидаемый результат

В результате выполнения лабораторной работы система должна корректно функционировать даже при наличии сетевых помех и обеспечивать измерение качества соединения.

---

### 9.1 Работа клиента (диагностика сети)

При выполнении команды:

```cpp
> /ping 5
```

клиент должен:

1. Отправить 5 последовательных запросов `MSG_PING`;
2. Для каждого запроса:
    - измерить время отклика (RTT);
    - вычислить джиттер (начиная со второго пакета);
    - определить, был ли ответ получен.

Пример вывода:

```cpp
PING 1 → RTT=4.2ms    
PING 2 → RTT=5.1ms | Jitter=0.9ms    
PING 3 → RTT=12.8ms | Jitter=7.7ms    
PING 4 → timeout    
PING 5 → RTT=6.3ms | Jitter=0.0ms  
```

Требования:

- для каждого пакета должен выводиться результат;
- при отсутствии ответа должен отображаться `timeout`;
- джиттер вычисляется только при наличии предыдущего RTT.

---

### 9.2 Итоговая статистика

После выполнения команды:

```cpp
> /netdiag
```

клиент должен вывести агрегированную статистику:

```cpp
RTT avg : 7.1 ms    
Jitter  : 2.8 ms    
Loss    : 20.0%
```

где:

- `RTT avg` — среднее значение всех успешных RTT;
- `Jitter` — среднее значение джиттера;
- `Loss` — процент потерянных сообщений.

Дополнительно результат должен сохраняться в файл:

```cpp
net_diag_<nickname>.json
```

---

### 9.3 Работа клиента (повторная отправка)

При потере сообщения клиент должен:

1. Зафиксировать отсутствие `ACK`;
2. Запустить повторную отправку;
3. Ограничить количество попыток (например, до 3);
4. Отобразить процесс в логах.

Пример:

```cpp
[Transport][RETRY] send MSG_TEXT (id=41)    
[Transport][RETRY] wait ACK timeout    
[Transport][RETRY] resend 1/3 (id=41)    
[Transport][RETRY] ACK received (id=41)
```

Требования:

- повторная отправка должна происходить автоматически;
- при получении ACK сообщение считается доставленным;
- при превышении числа попыток сообщение считается недоставленным.

---

### 9.4 Работа сервера

Сервер должен корректно обрабатывать сообщения даже при включённой эмуляции помех.

Пример логов:

```cpp
[Transport][SIM] DROP (id=41)    
[Transport][RETRY] resend detected (id=41)    
[Application][ACK] process MSG_TEXT (id=41)    
[Transport][ACK] send MSG_ACK (id=41)
```

Пояснение:

1. `[SIM] DROP` — сообщение было искусственно потеряно;
2. `[RETRY] resend detected` — клиент повторно отправил сообщение;
3. `[Application][ACK] process` — сообщение обработано сервером;
4. `[ACK] send` — сервер отправил подтверждение.

---

### 9.5 Защита от дубликатов

Если одно и то же сообщение (одинаковый `msg_id`) приходит несколько раз, сервер должен:

- обработать его только один раз;
- игнорировать все последующие дубликаты.

Пример:

```cpp
[Application][ACK] process MSG_TEXT (id=41)    
[Application][DEDUP] duplicate ignored (id=41)
```

---

### 9.6 Итоговое поведение системы

В результате выполнения лабораторной работы должно наблюдаться:

- корректное измерение параметров сети (RTT, jitter, loss);
- устойчивость работы при потере сообщений;
- автоматическая повторная отправка при отсутствии ACK;
- отсутствие дублирования сообщений на сервере;
- корректное логирование всех этапов обработки.

Система должна демонстрировать стабильную работу даже при заданных параметрах:

```cpp
--drop > 0    
--delay > 0    
--corrupt > 0
```
